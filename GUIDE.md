# VPN Balancer — Руководство пользователя

> Автоматический балансировщик нагрузки для VPN-инфраструктуры на базе Remnawave + Prometheus.
> Следит за здоровьем нод, выводит плохие из пула и возвращает хорошие, уведомляет в Telegram.

---

## Как это работает (за 30 секунд)

```
Prometheus  ←──  node_exporter (на каждой ноде)
                 ping_metrics.sh  (пинг → 77.88.8.8, каждые 5 мин)
                 speedtest.sh     (скорость канала, каждые 8 ч)

balancer.py ──→  читает метрики из Prometheus
            ──→  считает score (0.0 = отлично, 1.0 = мертво)
            ──→  управляет тегом хоста в Remnawave (/api/hosts)
            ──→  шлёт уведомления в Telegram
```

Каждые **2 минуты** балансировщик проверяет каждую ноду:
- score > **0.75** → нода выводится из пула (тег снимается)
- score < **0.55** → нода возвращается в пул (тег ставится)

## Установка — одна команда

Зайди на **сервер панели** (где установлен Remnawave) и выполни:

```bash
bash <(curl -4 -Ls "https://raw.githubusercontent.com/FnoUp/balancer_installer/master/setup.sh")
```

Скрипт задаст вопросы — заполни по таблице ниже:

| Что спросит | Где взять |
|-------------|-----------|
| Домен панели Remnawave | твой домен, например `panel.example.com` |
| Токен Remnawave | Панель → Admin → API Keys |
| Токен Telegram бота | от `@BotFather` |
| TG Chat ID | один общий чат для всех уведомлений — правой кнопкой на группе → Copy link → цифры. Enter принимает значение по умолчанию, зашитое в `setup.sh` |
| Topic ID — Ошибки/критика | топик внутри этого чата — правой кнопкой на топике → Copy link → цифры. Enter = дефолт из скрипта |
| Topic ID — Предупреждения | аналогично, Enter = дефолт из скрипта (системные сбои: Prometheus/API/полное падение сервиса) |
| Topic ID — Отчёты | аналогично |
| Topic ID — Метрики | аналогично, Enter = тот же топик, что Ошибки/критика |
| Название балансировщика | произвольное, отображается в TG-уведомлениях |

> Чат один и тот же для всех категорий — различаются только топики внутри него.

После установки панели нужно поставить агентов на **каждую ноду** — скрипт предложит это сделать автоматически (нужен SSH-доступ с панели к ноде).


**На каждой ноде (`<IP_НОДЫ>` — твои VPS с Xray):**
```

/var/lib/node_exporter/textfile_collector/vpn_metrics.prom  ← результаты
```

## Добавить новую ноду

```bash
# Зайди на сервер панели:
ssh root@<IP_ПАНЕЛИ>   # замени на реальный IP панели

python3 /opt/vpn-balancer/add_node.py
```

Скрипт спросит:

| Поле | Где взять |
|------|-----------|
| Название ноды | любое (fr1, de1, fi1) |
| IP адрес ноды | IP твоего VPS |
| Локация | france / finland / germany / … |
| Сетевой интерфейс | обычно `eth0` (проверь `ip link` на ноде) |
| UUID хоста Remnawave | Панель → Hosts → нужный хост → UUID |

Скрипт автоматически:
- Добавит цели в Prometheus
- Добавит ноду в список балансировщика
- Перезапустит сервисы
- Отправит уведомление в Telegram

## Автодобавление ноды (полная автоматизация)

Пункт меню `balancer` → **4) Автодобавление ноды**. Рассчитан на ноду,
у которой **config-profile и Reality-ключи уже настроены** (вручную или
заранее) — скрипт их не трогает и не создаёт новых, только подключает
клиентскую сторону. В отличие от `add_node.py` (ручной ввод IP/UUID),
этот пункт вообще не требует данных о ноде на входе — он сам:

1. Показывает список нод, уже подключённых к Remnawave (`GET /api/nodes`)
   с пометкой, есть ли у них config-profile — просто выбираешь номер,
   IP/UUID вводить не нужно. Если у выбранной ноды профиля нет — скрипт
   остановится с понятной ошибкой (сначала настрой её вручную).
2. Проверяет SSH-доступ с панели на ноду (`root@<адрес ноды>`) — если
   ключа нет, скажет выполнить `ssh-copy-id root@<адрес>` и остановится,
   ничего не поменяв.
3. Берёт уже привязанный к ноде config-profile и его VLESS-инбаунд
   напрямую из `GET /nodes` (`configProfile.activeConfigProfileUuid` +
   `activeInbounds`) — если инбаундов несколько (например, на
   бридж-нодах VLESS + shadowsocks), берёт именно VLESS.
4. Спрашивает только то, что нельзя вывести автоматически: пул
   (BALANCER / BALANCER_WIFI / BALANCER_MOBILE) и, если не получилось
   определить сетевой интерфейс автоматически, сам интерфейс.
5. Сам определяет сетевой интерфейс ноды: резолвит домен ноды через DNS,
   заходит по SSH и ищет интерфейс с таким же IP. Если получилось
   однозначно — не спрашивает вообще. Если на ноде несколько адресов
   (NAT/floating IP) — уточнит вручную (как в `add_node.py`).
6. Создаёт реальный Host (флаг+страна, fingerprint firefox, привязан к
   уже существующему инбаунду ноды) и отдельный виртуальный
   хост-держатель для клиентского RU-обхода (`<страна> (device-route
   RU)`, на общем config-profile `VirtualHost` — тоже уже существующем,
   не создаётся заново), плюс JSON-шаблон подписки для него.
7. Регистрирует ноду в `balancer.py` — как обычно в `add_node.py`.

Если что-то сломалось на середине — скрипт ничего сам не откатывает и не
удаляет, а печатает UUID всех уже созданных объектов Remnawave, чтобы
доразобраться вручную через панель.

## Обновить скрипт (панель/нода)

Отдельный пункт меню (панель — **11**, нода — **7**), который явно
спрашивает "Панель или Нода?" и подтягивает свежие файлы для выбранной
роли — не полагаясь на автоопределение по `/etc/vpn-balancer/config`.
Полезно, если нужно обновить роль, отличную от той, что определилась
автоматически, или просто хочется явного контроля. Логика обновления
та же, что и в пункте 1 (панель) / пункте 5 (нода) — это просто ещё один
вход в неё.

## Настройки балансировщика

Все переменные в `/opt/vpn-balancer/balancer.py` (секция констант вверху файла).
После изменений: `systemctl restart vpn-balancer`.

### Пороги score
```python
SCORE_BAD  = 0.75   # выше → нода выводится из пула
SCORE_GOOD = 0.55   # ниже → нода возвращается в пул
```
*Гистерезис 0.75/0.55 предотвращает «мерцание» — нода не прыгает туда-обратно каждые 2 минуты.*

### Критические пороги (→ score = 1.0 принудительно)
```python
CPU_CRITICAL = 90.0   # % — CPU выше этого = форс-мажор
RAM_CRITICAL = 90.0   # % — RAM выше этого = форс-мажор
MAX_PING_MS  = 300.0  # мс — пинг выше этого = форс-мажор
MAX_USERS    = 100    # юзеров — переполнение (алерт, не форс-мажор)
```

### Расчёт capacity (потолок канала)
```python
MIN_SPEEDTEST     = 100.0   # Мбит/с — ниже → штраф capacity=50
CAPACITY_FLOOR    = 50.0    # штраф при медленном/без данных speedtest
CAPACITY_GRACE    = 100.0   # для новых нод (<24ч, нет speedtest)
NODE_GRACE_PERIOD = 86400   # 24ч = 86400 секунд
```

| Условие | capacity |
|---------|----------|
| speedtest ≥ 100 Мбит/с | = speedtest (реальный потолок) |
| speedtest < 100 Мбит/с | = 50 (штраф) |
| нет speedtest + нода < 24ч | = 100 (grace, не штрафуем новую) |
| нет speedtest + нода ≥ 24ч + есть tx_p95 | = tx_p95 (реальный трафик) |
| нет данных | = 50 (штраф) |

### Веса компонентов score
```python
W_PING = 0.25   # пинг к 77.88.8.8
W_BW   = 0.50   # текущий tx / capacity  ← самый важный
W_USER = 0.15   # активные юзеры / 100
W_CPU  = 0.07   # загрузка CPU
W_RAM  = 0.03   # загрузка RAM
```
*Сумма весов = 1.0. score ∈ [0.0, 1.0], чем выше — тем хуже.*

### Интервалы
```python
CHECK_INTERVAL = 120    # секунды между проверками
DIGEST_HOUR    = 9      # час UTC для отправки дайджеста
ALERT_COOLDOWN = 1800   # 30 мин между повторными алертами одного типа
```

## Уведомления

Каждое уведомление начинается с первой строки: `⚙️ <Название балансировщика>` — чтобы сразу было видно откуда пришло.

Один чат, топики настраиваются при установке. По умолчанию Метрики и Критика используют **один и тот же топик** (`<ID_ОШИБОК>`), Предупреждения — отдельный (`<ID_ПРЕДУПРЕЖДЕНИЙ>`), Отчёты — тоже отдельный (`<ID_ОТЧЁТОВ>`). При желании можно развести Метрики на свой топик — спросится отдельно при установке.

### 📊 Метрики (топик `<ID_ОШИБОК>` по умолчанию)
| # | Уведомление | Когда |
|---|-------------|-------|
| 01 | 🚀 Балансировщик запущен | при старте сервиса |
| 02 | 📡 Пинг недоступен | 77.88.8.8 не отвечает |
| 03 | 📡 Пинг высокий | > 300 мс |
| 04 | ✅ Пинг восстановился | после 02 или 03 |
| 05 | ⚠️ CPU критично | ≥ 90% |
| 06 | ✅ CPU восстановился | после 05 |
| 07 | ⚠️ RAM критично | ≥ 90% |
| 08 | ✅ RAM восстановился | после 07 |
| 09 | 🐢 Нода медленная | speedtest < 100 Мбит/с |
| 10 | 👥 Нода переполнена | юзеров ≥ 100 |
| 11 | 🔴 Нода выведена из пула | score > 0.75 |
| 12 | 🟢 Нода возвращена в пул | score < 0.55 |
| 13 | 🟢 Новая нода подключена | после add_node.py |

### 🚨 Критика (топик `<ID_ОШИБОК>`)
| # | Уведомление | Когда |
|---|-------------|-------|
| 14 | ⚠️ Деградация — последняя нода перегружена | score > 0.75, нода одна — **выводить нельзя**, пользователи ещё в сети |

### ❗ Предупреждения (топик `<ID_ПРЕДУПРЕЖДЕНИЙ>`, по умолчанию отдельный топик)
Системные сбои — не про конкретную ноду, а про сам мониторинг/API:

| # | Уведомление | Когда |
|---|-------------|-------|
| 15 | ❌ Prometheus недоступен | не читаются метрики |
| 16 | ❌ Ошибка API Remnawave | не удалось обновить тег |
| 17 | 🚨 Сервис недоступен — все ноды упали | пул пуст, подключения невозможны |

> **Разница между 14 и 17:** 14 — сервис деградировал (нода в пуле, юзеры ещё подключены), 17 — сервис полностью упал (0 нод в пуле).

### 📋 Отчёты (топик `<ID_ОТЧЁТОВ>`)
| # | Уведомление | Когда |
|---|-------------|-------|
| 18 | 📊 Дайджест нод | ежедневно в 09:00 UTC |
| 19 | 📋 Лог за сутки | сразу после дайджеста |

Итого 19 типов уведомлений, три топика (Метрики+Критика, Предупреждения, Отчёты).

## Формула score

```
score = ping_score × 0.25
      + bw_score   × 0.50
      + user_score × 0.15
      + cpu_score  × 0.07
      + ram_score  × 0.03

где:
  ping_score = clamp(ping_ms / 300)
  bw_score   = clamp(tx_mbps / capacity)
  user_score = clamp(users / 100)
  cpu_score  = clamp(cpu_pct / 100)
  ram_score  = clamp(ram_pct / 100)
  clamp(x)   = min(1.0, max(0.0, x))
```

Если пинг недоступен / CPU ≥ 90% / RAM ≥ 90% — score = **1.0** (принудительно).

**Примеры:**
- Отличная нода: ping=50ms, tx=5/700Мбит/с, 3 юзера, cpu=3%, ram=34% → score ≈ **0.06**
- Нагруженная: ping=150ms, tx=45/50Мбит/с, 60 юзеров, cpu=70%, ram=75% → score ≈ **0.67**
- Мёртвая: ping недоступен → score = **1.0**

## Диагностика

### Проверить, что балансировщик работает
```bash
ssh root@<IP_ПАНЕЛИ>        # замени на IP сервера с Remnawave
systemctl status vpn-balancer
tail -f /var/log/vpn-balancer/balancer.log
```

### Посмотреть текущий score нод
```bash
grep "score=" /var/log/vpn-balancer/balancer.log | tail -20
# Вывод: France: score=0.056 users=3 | ping=52ms bw=1.3/712Мбит/с ...
```

### Проверить метрики Prometheus
```
# Открыть в браузере (замени IP):
http://<IP_ПАНЕЛИ>:9090/targets   ← все цели должны быть UP
http://<IP_ПАНЕЛИ>:9090/graph     ← ручные запросы

# Пример запроса — текущий пинг ноды:
vpn_node_ping_ms{instance="<IP_НОДЫ>:9100"}
```

### Проверить что node_exporter отдаёт метрики с ноды
```bash
# С сервера панели (замени IP):
curl http://<IP_НОДЫ>:9100/metrics | grep vpn_node
# Должны быть: vpn_node_ping_ms, vpn_node_ping_ok, vpn_node_capacity_mbps
```

### Принудительно запустить speedtest / пинг на ноде
```bash
ssh root@<IP_НОДЫ>     # замени на IP ноды
bash /etc/vpn-balancer/ping_metrics.sh
bash /etc/vpn-balancer/speedtest.sh
cat /var/lib/node_exporter/textfile_collector/vpn_metrics.prom
```

### Отправить тестовые уведомления
```bash
# На сервере панели:
ssh root@<IP_ПАНЕЛИ>

# Все 19 сценариев (тестовые данные):
python3 /tmp/sim_alerts.py

# Реальные данные из Prometheus:
python3 /tmp/real_alerts.py
```

## Частые проблемы

| Симптом | Причина | Решение |
|---------|---------|---------|
| `node_textfile_scrape_error 1` в метриках | vpn_metrics.prom недоступен node_exporter | `chmod 644 /var/lib/node_exporter/textfile_collector/vpn_metrics.prom` |
| Уведомления дублируются | двойное логирование | в systemd-сервисе должно быть `StandardOutput=journal` |
| `users=?` в логе | нет ответа от `/api/nodes` | проверь REMNAWAVE_TOKEN и REMNAWAVE_COOKIE в balancer.py |
| Speedtest 0 или нет | cron не запустился | `crontab -l` на ноде, проверь `/etc/vpn-balancer/speedtest.sh` |
| Score всегда ~0 | нет tx-трафика (нода пустая) | нормально, при росте нагрузки score вырастет |
| Нода не возвращается в пул | score выше 0.55 | посмотри метрики в Prometheus, найди узкое место |

## Структура данных нод в balancer.py

```python
NODES = [
    {
        "name":          "🇫🇷 France",        # имя в TG-уведомлениях
        "host_uuid":     "<UUID из Remnawave>", # Панель → Hosts → UUID
        "prom_instance": "<IP_НОДЫ>:9100",      # цель Prometheus (node_exporter)
        "ping_instance": "<IP_НОДЫ>",           # IP для blackbox probe
        "net_device":    "eth0",                # сетевой интерфейс (`ip link` на ноде)
    },
    # ... следующие ноды добавляются через add_node.py
]
```

После изменения этого файла вручную: `systemctl restart vpn-balancer`.
